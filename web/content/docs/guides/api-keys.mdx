---
title: API keys
description: Create org-scoped keys so your code can call the gateway.
---

The dashboard signs people in with a session cookie, but your own code needs a
credential it can carry on every request. That's an **API key**. You create keys
in the dashboard; the gateway checks them on each call.

## Create one

In the dashboard, on your organization's API keys page, create a key. You can set:

- a **name**, so you can tell keys apart and revoke the right one later,
- an optional **expiry**,
- an optional **rate limit**, and
- the **permissions** it carries (below).

The full key is shown **once**, at creation. Copy it then — you cannot read it
again, only revoke it and make a new one.

## Use it

Send the key as a Bearer token, or in the `x-api-key` header:

```bash
curl https://your-gateway/api/v1/sessions \
  -H "Authorization: Bearer wa_..."
# or
curl https://your-gateway/api/v1/sessions \
  -H "x-api-key: wa_..."
```

The same endpoints accept either an API key (for your code) or a user token (for
the dashboard) — you don't need different URLs.

## Keys are scoped to one organization

A key belongs to exactly **one** organization — the one it was created in — and it
can only touch that org's sessions, messages, webhooks, and events. There is no
account-wide key. If you work across several organizations, you create a separate
key in each.

This is the same boundary the dashboard enforces for people: a credential only
ever sees the resources of the org it belongs to.

## Permissions

A key carries a set of permissions, and the gateway checks them per action:

| Permission | Lets the key |
|---|---|
| `read` | List and read sessions, chats, contacts, messages. |
| `send` | Send messages and message actions (react, edit, revoke, forward). |
| `manage` | Create and change sessions and webhooks. |
| `events` | Open the [event stream](/docs/guides/receiving) and receive events. |

Give each key the least it needs — a key that only pushes notifications wants
`send`, not `manage`.

<Callout type="warn">
A key is a bearer credential: anyone who has it can act as that org until you
revoke it. Keep keys out of source control and client-side code, and revoke a
key the moment it might be exposed.
</Callout>
