title: Receive events
description: Read gateway events by holding open an HTTP stream, or have the gateway POST each one to a URL you run.

Whenever something happens on one of your sessions â€” a message arrives, a delivery
receipt comes back, someone votes on a poll, a session changes status â€” the gateway
produces an **event**. You read those events in one of two ways:

| Way | How it works | Use it when |
|---|---|---|
| **Stream** | You open one long-lived HTTP connection and the gateway writes events to it as they happen. | Your code can hold a connection open â€” a worker, a script, the dashboard. |
| **Webhooks** | The gateway sends an HTTP `POST` to a URL you run, once per event, and retries until it succeeds. | You have a public URL and would rather not hold a connection open. |

Both carry the **same event envelope** (the JSON wrapper described below), so the data
you receive is identical whichever you pick.

## The event envelope

Every event is one JSON object with the same outer fields:

```json
{
  "schema": "v1",
  "id": "evt_01J9...",
  "event": "message",
  "session": "sess_01J8...",
  "organization": "org_abc",
  "timestamp": 1719400000000,
  "payload": { }
}
```

| Field | What it is |
|---|---|
| `schema` | Envelope version. Currently `v1`. |
| `id` | Unique id for this event. Use it to drop duplicates (see webhooks below). |
| `event` | The event type â€” one of the names in the [catalog](#event-catalog) below. |
| `session` | The session the event belongs to. |
| `organization` | The organization that owns the session. |
| `timestamp` | When it happened, in epoch milliseconds (milliseconds since 1 Jan 1970). |
| `payload` | The type-specific data. Its shape depends on `event`. |

For every field of every payload type, with concrete examples, see
[Event shape](/docs/guides/event-shape).

## Stream the events

Open a `GET` to the events endpoint with your Bearer token. The response is
`application/x-ndjson` â€” NDJSON means newline-delimited JSON: the connection stays
open and the gateway writes one event per line, each line a single JSON object ending
in `\n`.

```bash
curl -N https://your-gateway/api/v1/events?events=* \
  -H "Authorization: Bearer $TOKEN"
```

Control what you receive with query parameters:

| Parameter | Effect |
|---|---|
| `events=*` | Subscribe to every event type. |
| `events=message,poll.vote` | Subscribe to only the named types (comma-separated). |
| `session={id}` | Follow one session. Omit it to receive events for every session your organization owns. |
| `since={lastEventId}` | Replay the events you missed since that id, then keep tailing live events. |

Two things to know about a live stream:

- **Heartbeat.** A `ping` event arrives about every 20 seconds. If you stop seeing
  pings, the connection is dead and you should reconnect.
- **Reconnecting after a drop.** Reconnect with `since={lastEventId}` set to the `id`
  of the last event you processed. The gateway replays everything after that id from
  its durable log, then resumes tailing. Replayed events look identical to live ones,
  so you cannot tell them apart â€” which is why deduping on `id` matters.

This is a plain chunked HTTP stream, **not** Server-Sent Events: there is no
`text/event-stream` content type and no `data:` framing. Read it with `fetch` and a
`ReadableStream`, and set the `Authorization` header the same way as any other call.
See [stream events](/docs/api/events/events).

<Callout type="info">
A dashboard JWT expires after about 5 minutes, but the stream only checks your token
when you connect. To keep going past expiry, refresh your token and reconnect with
`since={lastEventId}` â€” the replay stitches the gap so your view is never torn.
</Callout>

## Use webhooks instead

Register a webhook and the gateway delivers each matching event to your URL as a JSON
`POST`. You create and manage webhooks through the gateway API
([create a webhook](/docs/api/webhooks/createWebhook)); the dashboard shows the same
config read-only.

A webhook is configured with:

| Setting | What it does |
|---|---|
| URL | Where to send the `POST`. |
| events | Which event types to send: `["*"]` for all, or a list like `["message","poll.vote"]`. An empty list matches nothing. |
| session | Optional. Limit to one session. Leave it off to cover all the organization's sessions. |
| custom headers | Optional. Extra headers added to every delivery (applied last, so they can override the defaults below). |
| retry policy | Optional. How many times and how often to retry a failed delivery. |

### Delivery headers

Every `POST` carries these headers:

| Header | Purpose |
|---|---|
| `Content-Type: application/json` | The body is the JSON event envelope. |
| `X-Webhook-Request-Id` | The event `id`. Use it to drop duplicate deliveries. |
| `X-Webhook-Timestamp` | When the gateway sent this delivery (epoch milliseconds). |
| `X-Webhook-Hmac` | Signature of the body. Present only when the webhook has a secret. |
| `X-Webhook-Hmac-Algorithm` | The signature algorithm. Always `sha512`. |

### Before you trust a delivery

- **Check the signature.** When the webhook has a secret, compute the HMAC-SHA512 of
  the exact raw request body using your secret, encode it as lowercase hex, and compare
  it to `X-Webhook-Hmac`. If they differ, reject the request. (HMAC is a keyed hash:
  only someone holding the secret can produce a matching value, so a match proves the
  body came from the gateway and was not altered.)
- **Drop duplicates.** A retried delivery repeats the same event `id`
  (`X-Webhook-Request-Id`). Keep track of ids you have already handled and skip repeats.

The gateway retries a failed delivery with growing gaps between attempts. Once a
delivery is accepted (a 2xx response) it stops; once it runs out of retry attempts it
is marked **dead** and not tried again.

## Event catalog

Every event type the gateway emits. Subscribe with `events=*` for all of them, or name
the ones you want.

| Event | Fires when |
|---|---|
| `session.status` | A session changes status (for example `working`, `logged_out`). |
| `auth.qr` | A new QR code is available to scan. |
| `auth.code` | A pairing code is available. |
| `message` | A message comes in. |
| `message.from_me` | A message was sent from this number, including from another linked device. |
| `message.status` | A delivery receipt updates a sent message (sent, then delivered, then read). |
| `message.reaction` | Someone reacts to a message. |
| `message.edited` | A message is edited. |
| `message.revoked` | A message is deleted for everyone. |
| `poll.vote` | Someone votes on a poll. |
| `presence.update` | A contact goes online or offline, or starts typing. |
| `group.update` | A group's metadata changes (name, settings). |
| `group.participant` | Someone joins, leaves, or changes role in a group. |
| `chat.update` | A chat changes (archived, pinned, read state). |
| `contact.update` | A contact's name or details change. |
| `call.incoming` | A call comes in. |
| `newsletter.update` | A followed channel updates. |

<Callout type="info">
**Media is metadata only.** A message with an attachment arrives with `hasMedia: true`
and `media: null` â€” you see that it has media and what type it is, but this release
does not download the file.
</Callout>
