Every event the gateway emits — on the NDJSON stream and on webhooks — is the same
JSON object: a versioned **envelope** that wraps a type-specific **payload**. This
page is the reference for that shape, with a fully-populated example for every
event type and every state it can take. For how to subscribe, see
[Receive events](/docs/guides/receiving); for the endpoint contract, see the
[stream events API](/docs/api/events/events).

## The envelope

```json
{
  "schema": "v1",
  "id": "evt_01J9XYZABCDEF0123456789",
  "event": "message",
  "session": "sess_01J8ABCDEF0123456789",
  "organization": "org_abc",
  "timestamp": 1719400000000,
  "payload": {}
}
```

| Field | Type | What it is |
|---|---|---|
| `schema` | string | Envelope version. Currently `v1`. Branch on this if it ever changes. |
| `id` | string | The event id, a ULID with an `evt_` prefix. Stable and unique. Use it to dedupe and to resume the stream. |
| `event` | string | The event type — one of the [catalog](#event-catalog) below. Tells you how to read `payload`. |
| `session` | string | The session the event belongs to (`sess_…`). |
| `organization` | string | The organization that owns the session. Every event is scoped to one org. |
| `timestamp` | integer | When the gateway emitted it, in **epoch milliseconds**. |
| `payload` | object | Type-specific data. Its shape depends on `event` — see each entry below. |

The same envelope rides both transports, so your handler can be written once:

- **Stream** — `GET /api/v1/events`, `application/x-ndjson`, one envelope per line. A
  `ping` event arrives about every 20s; resume after a drop with `since={lastEventId}`.
- **Webhooks** — the gateway `POST`s the envelope as JSON. The envelope `id` is also
  sent as `X-Webhook-Request-Id`; dedupe on it. Deliveries are HMAC-signed
  (`X-Webhook-Hmac`, `X-Webhook-Hmac-Algorithm: sha512`) and retried.

## Subscribe to the stream

```bash
curl -N "https://your-gateway/api/v1/events?events=*&session=sess_01J8ABCDEF0123456789" \
  -H "Authorization: Bearer $TOKEN"
```

- `events=*` is every type; pass a comma list (`events=message,poll.vote`) to narrow.
- `session=` filters to one session; omit it to receive every session your org owns.
- After a disconnect, reconnect with `since={lastEventId}` — the gateway replays what
  you missed from the event log, then tails live.

See the [stream events API](/docs/api/events/events) for the full endpoint detail.

## Event catalog

Seventeen event types, grouped below. Every example is a complete envelope you can
copy. Optional fields are omitted from the wire when empty; each example calls out
which fields are optional or nullable.

| Event | Fires when |
|---|---|
| [`session.status`](#sessionstatus) | A session changes connection state. |
| [`auth.qr`](#authqr) | A new QR code is ready to scan. |
| [`auth.code`](#authcode) | Pairing succeeded. |
| [`message`](#message) | An inbound message arrives. |
| [`message.from_me`](#messagefrom_me) | A message is sent from this number (incl. another linked device). |
| [`message.status`](#messagestatus) | A receipt updates a sent message. |
| [`message.reaction`](#messagereaction) | Someone reacts to a message. |
| [`message.edited`](#messageedited) | A message is edited. |
| [`message.revoked`](#messagerevoked) | A message is deleted for everyone. |
| [`poll.vote`](#pollvote) | Someone votes on a poll. |
| [`presence.update`](#presenceupdate) | A contact goes online/offline or starts typing. |
| [`group.update`](#groupupdate) | A group's metadata changes. |
| [`group.participant`](#groupparticipant) | Someone joins, leaves, or changes role. |
| [`chat.update`](#chatupdate) | A chat changes (currently profile-picture changes). |
| [`contact.update`](#contactupdate) | A contact's name or details change. |
| [`call.incoming`](#callincoming) | An incoming call. |
| [`newsletter.update`](#newsletterupdate) | A followed channel updates. |

<Callout type="info">
**Media is metadata only.** A message with an attachment arrives with `hasMedia: true`
and `media: null`. You learn that the message has media (and, internally, its type) but
this release never downloads the file.
</Callout>

## Messages

The `message` and `message.from_me` events share one flat payload, discriminated by
`type`. `media` is **always `null`** on the wire; `hasMedia` tells you whether an
attachment exists.

### `message`

An inbound message. Pick a tab per content type.

<Tabs items={["text", "media", "location", "contact", "poll"]}>

<Tab value="text">

A plain text message. A reply also carries `quotedMessageId`; a message with mentions
carries a `mentions` JID list (both shown here).

```json
{
  "schema": "v1",
  "id": "evt_01J9MSGTEXT0000000000001",
  "event": "message",
  "session": "sess_01J8ABCDEF0123456789",
  "organization": "org_abc",
  "timestamp": 1719400010000,
  "payload": {
    "waMessageId": "3EB0A1B2C3D4E5F6A7B8",
    "chatJid": "120363012345678901@g.us",
    "senderJid": "6281234567890@s.whatsapp.net",
    "senderLid": "123456789012345@lid",
    "fromMe": false,
    "type": "text",
    "body": "@628999 are we still on for tomorrow?",
    "quotedMessageId": "3EB0FEDCBA9876543210",
    "mentions": ["628999@s.whatsapp.net"],
    "hasMedia": false,
    "media": null,
    "timestamp": 1719400010000,

```

</Tab>

Image / video / audio / document / sticker. `type` is the specific media kind,

when present, rides in `body`.

```json
{
  "schema": "v1",
  "id": "evt_01J9MSGMEDIA000000000001",
  "event": "message",
  "session": "sess_01J8ABCDEF0123456789",
  "organization": "org_abc",
  "timestamp": 1719400011000,
  "payload": {
    "waMessageId": "3EB0A1B2C3D4E5F6A7B9",
    "chatJid": "6281234567890@s.whatsapp.net",
    "senderJid": "6281234567890@s.whatsapp.net",
    "fromMe": false,
    "type": "image",
    "body": "here's the receipt",
    "hasMedia": true,
    "media": null,
    "timestamp": 1719400011000,
    "pushName": "Alex"
  }
}
```

<Callout type="info">
`type` is one of `image`, `video`, `audio`, `document`, `sticker`. The file is never
downloaded — `media` stays `null` even though `hasMedia` is `true`.
</Callout>

</Tab>

```json

  "event": "message",
```

```json
  "organization": "org_abc",
  "timestamp": 1719400014000,
  "payload": {
    "waMessageId": "3EB0A1B2C3D4E5F6A7BC",
    "chatJid": "120363012345678901@g.us",
    "senderJid": "6281234567890@s.whatsapp.net",
    "fromMe": false,
    "type": "poll",
    "body": "Lunch on Friday?",
    "hasMedia": false,
    "media": null,

      "name": "Lunch on Friday?",
```

</Tab>

</Tabs>

**Common fields** (every `message` / `message.from_me` payload)

- `waMessageId` (string, required) — the WhatsApp message id; the target other events reference.
- `chatJid` (string, required) — the chat (DM `@s.whatsapp.net`, group `@g.us`, etc.).
- `senderJid` (string, optional) — phone-number JID of the sender.
- `senderLid` (string, optional) — sender's LID identity, when the addressing uses it.
- `fromMe` (boolean, required) — `false` for `message`; `true` for `message.from_me`.
- `type` (string, required) — `text`, `image`, `video`, `audio`, `document`, `sticker`, `location`, `contact`, `poll`.
- `body` (string, optional) — text / caption / poll name / contact display name.
- `quotedMessageId` (string, optional) — set when the message is a reply.
- `mentions` (string[], optional) — mentioned JIDs.
- `hasMedia` (boolean, required) — `true` when an attachment exists; `media` is still `null`.
- `media` (object|null) — always `null` in v1.
- `timestamp` (integer, required) — epoch-ms.
- `pushName` (string, optional) — the sender's display name.

### `message.from_me`

Identical shape to [`message`](#message), emitted when the message was sent from this
number — including from another linked device. The only difference is `fromMe: true`.

```json
{

  "id": "evt_01J9MSGFROMME00000000001",
  "event": "message.from_me",
  "session": "sess_01J8ABCDEF0123456789",
  "organization": "org_abc",
  "timestamp": 1719400015000,
  "payload": {
    "waMessageId": "3EB0A1B2C3D4E5F6A7BD",
    "chatJid": "6281234567890@s.whatsapp.net",
    "senderJid": "6289876543210@s.whatsapp.net",
    "fromMe": true,
    "type": "text",
    "body": "yes, see you at noon",
    "hasMedia": false,
    "media": null,
    "timestamp": 1719400015000,
    "pushName": "You"
  }
}
```

### `message.status`

A receipt updates one or more sent messages. One receipt can cover several messages,
so `messageIds` is a list.

<Callout type="info">
Receipts from WhatsApp only ever raise the status to `delivered`, `read`, or `played`.
The earlier lifecycle values — `pending`, `sent`, `failed` — are set internally when
the gateway accepts and dispatches your outbound send; they describe the same
`status` field but are not driven by an inbound receipt.
</Callout>

## Polls

### `poll.vote`

Someone voted on a poll. `targetId` is the poll-creation message being voted on. The
selected options arrive as **encrypted** `selectedHashes` rather than plaintext labels.

```json
{
  "schema": "v1",
  "id": "evt_01J9POLLVOTE0000000000001",
  "event": "poll.vote",
  "session": "sess_01J8ABCDEF0123456789",
  "organization": "org_abc",
  "timestamp": 1719400024000,
  "payload": {
    "waMessageId": "3EB0A1B2C3D4E5F6A7C5",
    "chatJid": "120363012345678901@g.us",
    "senderJid": "6281234567890@s.whatsapp.net",
    "fromMe": false,
    "type": "poll_vote",
    "targetId": "3EB0A1B2C3D4E5F6A7BC",
    "selectedHashes": ["a1b2c3d4e5f6a7b8", "d4e5f6a7b8c9d0e1"],
    "hasMedia": false,
    "media": null,
    "timestamp": 1719400024000,
    "pushName": "Alex"
  }
}
```

**Fields**

- `targetId` (string) — the poll-creation message's `waMessageId`.
- `selectedHashes` (string[]) — encrypted hashes of the chosen options (not plaintext labels).
- Plus the common message fields (`waMessageId`, `chatJid`, `senderJid`, `fromMe`, `timestamp`, `pushName`).

## Presence, chats & contacts

<Accordions type="single">

<Accordion title="presence.update — online/offline or typing" id="presenceupdate">
