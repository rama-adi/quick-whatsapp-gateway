---
title: Connect a number
description: Create a session and link a phone with a QR code or pairing code.
---

A **session** is one linked WhatsApp number. You pair it once — by scanning a QR
code or entering a pairing code on the phone — and the gateway keeps the
connection alive after that, reconnecting on its own if it drops.

## 1. Create the session

In the dashboard, on your organization's sessions page, create a session and give
it a label. Programmatically it is a `POST /sessions` with an optional label and a
`start` flag — see [create a session](/docs/api/sessions/createSession).

A fresh session starts in status `scan_qr_code`: it is waiting for a phone.

## 2. Link the phone

Pick whichever the phone makes easier. Both link the same way WhatsApp's own
"Linked devices" feature does.

### Scan a QR code

The session exposes a QR string that refreshes about every 20 seconds. The
dashboard renders it for you and swaps it as it rotates. On the phone: **WhatsApp
→ Linked devices → Link a device**, then scan.

Fetch the current code yourself with [get the QR code](/docs/api/sessions/sessionQR)
(`?format=image` returns a PNG instead of the raw string).

### Use a pairing code

If scanning is awkward, ask for an 8-character pairing code tied to the phone
number, then type it into the phone under **Link a device → Link with phone number
instead**:

```bash
curl -X POST https://your-gateway/api/v1/sessions/{id}/pairing-code \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"phone":"628123456789"}'
```

See [request a pairing code](/docs/api/sessions/sessionPairingCode). The number is
in international format with no `+` or spaces.

## 3. Confirm it linked

Once the phone accepts, the session moves to status `working`. These are all the
statuses a session moves through:

| Status | Meaning |
|---|---|
| `starting` | Connecting to WhatsApp. |
| `scan_qr_code` | Waiting for you to scan the QR or enter the pairing code. |
| `working` | Linked and live — ready to send and receive. |
| `stopped` | Stopped on purpose; reconnects when you start it. |
| `failed` | Connection failed; the gateway stopped retrying. |
| `logged_out` | The link was removed (from the phone or by a ban); needs re-pairing. |

The dashboard updates these live. Over the API you can [poll the session](/docs/api/sessions/getSession),
or subscribe to `session.status` on the [event stream](/docs/guides/receiving).

## The admin number

A deployment can name one number as its **admin number** through the gateway's
`WHATSAPP_ADMIN_NUMBER` setting. On boot, if that number is not already linked,
the gateway creates its session, prints a pairing code to its console, and shows
the same code on the admin surface — so the operator can pair it without a browser.

The admin number is otherwise an ordinary session: it sends and receives like any
other. It also reserves a private command channel — messages it receives that
start with the admin command prefix (`am` by default) are handled internally and
are **not** stored, emitted as events, or counted against rate limits. In this
release that channel is reserved but does nothing; commands arrive in a later
version.
