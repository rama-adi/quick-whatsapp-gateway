title: Repo layout

The repo is a monorepo: the gateway is at the root, the frontend is under `web/`.

```
.
├── cmd/server/main.go     gateway entrypoint (also: server migrate up|down)
├── internal/              gateway packages (Go)
├── migrations/            golang-migrate — WA app-data schema only
├── web/                   frontend — TanStack Start + better-auth
├── deploy/                Dockerfile · Dockerfile.web · compose files · .env.example
├── docs/                  openapi.yaml · specs/*.md · mvp-progress.md
└── masterplan-mvp.md      full v2 design
```

## Gateway packages (`internal/`)

| Package | Responsibility |
|---|---|
| `config/` | env parsing and defaults |
| `authz/` | the trust seam — JWKS+JWT verify, api-key verify, the two-acceptor middleware, capability gates, CORS |
| `controlbus/` | subscribes to `ctrl:*`; evicts the api-key cache and drops live streams on revocation |
| `domain/` | core entities, enums, IDs, errors — no I/O |
| `http/` | chi router, middleware, and the REST handlers (no embedded SPA) |
| `httpx/` | request decode, JSON responses, pagination helpers |
| `wa/` | the WhatsApp engine — `manager`, `session`, the SQLite keystore, and `inbound/` · `outbound/` · `events/` pipelines |
| `store/` | MySQL repositories, all keyed by `organization_id` |
| `stream/` | the NDJSON event stream — registry, filter, publisher |
| `queue/` | asynq job queue on the work Redis (outbound send, webhook delivery, retention) |
| `service/` | wires the pieces together — the application layer between handlers and stores |
| `crypto/` | AES-GCM for webhook secrets at rest, api-key hashing |

`internal/` from v1 is gone: the embedded auth (`internal/auth`), the embedded
SPA, and the custom MySQL whatsmeow store. `internal/authz` replaced them.

## Frontend (`web/app/`)

```
web/app/
├── routes/                file-based route tree
│   ├── __root.tsx         root document + providers
│   ├── index.tsx          role-routed entry (→ login / admin / user)
│   ├── _auth.tsx + _auth/*    public auth surface (login, register, 2fa)
│   ├── _app.tsx + _app/*      authenticated shell; resolves the session in beforeLoad
│   │   ├── admin/*         super_admin surface (sessions, tenants, monitor, pairing)
│   │   └── user/*          user workspace (sessions, keys, webhooks) + org switcher,
│   │                        plus the chat viewer and contacts routes
│   ├── api/auth.$.ts       better-auth catch-all (mounts /api/auth/*)
│   ├── api/search.ts       docs search endpoint
│   └── docs/$.tsx          the Fumadocs catch-all that renders these pages
├── components/
│   ├── ui/                 ported shadcn primitives
│   └── shell/              app chrome
└── lib/
    ├── auth/               better-auth client + server config, session, token, middleware
    ├── api/                gateway client, generated OpenAPI types, React Query hooks, query keys
    ├── db/                 Drizzle: auth-schema.ts (generated) + wa.ts (read-only WA mirror)
    ├── events/             NDJSON stream consumer, eventBus, cacheBridge reducer
    ├── server/             control-bus publisher (publish-only ctrl:*)
    ├── source.ts           Fumadocs content loader
    └── openapi.ts          OpenAPI page config for the reference docs
```

Conventions in the route tree:

- Files prefixed `-` are colocated helpers **excluded** from the route tree (e.g. `-viewer-data.ts`, `-message-bubble.tsx`).
- `_auth` and `_app` are pathless layout routes — `_app.tsx` resolves the session server-side in `beforeLoad` and puts it on the route context.
- `app/lib/**` and `app/components/{ui,shell}/**` are the shared boundary reused across routes.

## Docs content (`web/content/docs/`)

The Fumadocs MDX site lives here, served by the `docs/$.tsx` route:

```
web/content/docs/
├── index.mdx              docs home
├── guides/                task-based walkthroughs (pairing, sending, webhooks)
├── api/                   GENERATED OpenAPI reference — do not hand-edit
└── dev/                   these contributor docs
```

The `api/` tree is regenerated from the OpenAPI contract — see
[regenerating code](/docs/dev/regenerating-code). Full v2 layout: masterplan §16.
