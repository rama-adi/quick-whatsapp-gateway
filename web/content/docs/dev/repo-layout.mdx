This is one repo holding two codebases: the Go backend at the root (which runs as two processes — the router and the gateway) and the frontend (TanStack Start) under `web/`.

```
.
├── cmd/router/main.go     router entrypoint — the public API service
├── cmd/server/main.go     gateway entrypoint (also: server migrate up|down)
├── internal/              backend packages (Go), shared by the router and gateway
├── migrations/            golang-migrate — WA app-data schema only
├── web/                   frontend — TanStack Start + better-auth
├── deploy/                Dockerfile · Dockerfile.router · Dockerfile.web · compose files · .env.example
├── docs/                  openapi.yaml · specs/*.md · mvp-progress.md
└── masterplan-mvp.md      full v2 design
```

## Backend packages (`internal/`)

These are shared by both Go processes. Some run only on the router, some only on
the gateway; the "Runs on" column says which.

| Package | Runs on | What it does |
|---|---|---|
| `config/` | both | Reads configuration from environment variables and applies defaults. |
| `domain/` | both | Core entities, enums, IDs, and errors. No I/O. |
| `router/` | router | Public API routing: finds the session's gateway, enforces org isolation, forwards requests, places new sessions, serves realtime WebSockets, and reads the gateway registry. |
| `authz/` | router | Verifies who is calling — better-auth JWTs against the JWKS, and better-auth api-keys — and resolves the caller's permissions. A JWT is a signed token proving identity; JWKS is the public-key set used to verify its signature. |
| `assertion/` | both | The router→gateway internal assertion. The router-side `Minter` signs a per-request Ed25519 token; the gateway-side `Verifier` checks its signature, binding (method/path/body), and one-time nonce before trusting it. |
| `controlbus/` | router | Subscribes to the Redis `ctrl:*` channels. On a revocation message, evicts the api-key from the router's cache. |
| `http/` | gateway | The chi router, middleware (including the assertion verifier), and REST handlers (huma operations). No embedded frontend. |
| `httpx/` | both | HTTP helpers: request decoding, JSON responses, pagination, error mapping. |
| `wa/` | gateway | The WhatsApp engine. Holds the `manager` and `session` lifecycle, the SQLite keystore (`store/`), and the `inbound/`, `outbound/`, and `events/` pipelines. |
| `store/` | both | MySQL repositories (sessions, gateways registry, messages, …). Every query is keyed by `organization_id`. |
| `stream/` | both | The realtime event stream: the durable replay log, filtering, and the transport-agnostic pump the router's WebSocket drives. (The old NDJSON wire framing was removed.) |
| `queue/` | gateway | The asynq job queue on the work Redis. Runs outbound sends, webhook delivery, and retention jobs. |
| `webhooks/` | gateway | Outbound webhooks: matches events to configured endpoints, persists a pending delivery, then POSTs it with HMAC-signed headers, retries, and dead-letters exhausted deliveries. |
| `crypto/` | both | AES-GCM encryption for webhook secrets at rest, plus api-key hashing. |
| `service/` | gateway | The layer between handlers and stores. Wires the pieces together and holds the application logic. |

What was removed from v1: the embedded auth package, the embedded React SPA, and the custom MySQL whatsmeow keystore. `internal/authz` and the gateway-local SQLite keystore replaced them.

## Frontend (`web/app/`)

```
web/app/
├── routes/                file-based route tree
│   ├── __root.tsx         root document + providers
│   ├── index.tsx          entry route; redirects by role (login / admin / user)
│   ├── _auth.tsx + _auth/*    public auth pages (login, register, 2fa)
│   ├── _app.tsx + _app/*      authenticated shell; resolves the session in beforeLoad
│   │   ├── admin/*         super_admin pages (sessions, tenants, monitor, pairing)
│   │   └── user/*          user workspace (sessions, keys, webhooks, org switcher,
│   │                        chat viewer, contacts)
│   ├── api/auth.$.ts       better-auth catch-all (mounts /api/auth/*)
│   ├── api/search.ts       docs search endpoint
│   └── docs/$.tsx          Fumadocs catch-all that renders the docs pages
├── components/
│   ├── ui/                 ported shadcn primitives
│   └── shell/              app chrome
└── lib/
    ├── auth/               better-auth client + server config, session, token, middleware
    ├── api/                gateway client, generated OpenAPI types, React Query hooks, query keys
    ├── db/                 Drizzle: auth-schema.ts (generated) + wa.ts (read-only WA mirror)
    ├── events/             realtime WebSocket client, eventBus, cacheBridge reducer
    ├── server/             control-bus publisher (publishes ctrl:* only)
    ├── source.ts           Fumadocs content loader
    └── openapi.ts          OpenAPI page config for the reference docs
```

How the route tree works:

- A `-` prefix marks a colocated helper that is **excluded** from the route tree (for example `-contacts-data.ts`, `-message-bubble.tsx`).
- `_auth` and `_app` are pathless layout routes — they add no URL segment. `_app.tsx` resolves the session on the server in `beforeLoad` and puts it on the route context.
- `app/lib/**` and `app/components/{ui,shell}/**` are the shared code reused across routes.

## Docs content (`web/content/docs/`)

The Fumadocs MDX site lives here. The `docs/$.tsx` route serves it.

```
web/content/docs/
├── index.mdx              docs home
├── guides/                task-based walkthroughs (pairing, sending, webhooks)
├── api/                   GENERATED OpenAPI reference — do not hand-edit
└── dev/                   these contributor docs
```

The `api/` tree is regenerated from the OpenAPI contract — see
[regenerating code](/docs/dev/regenerating-code). The full v2 layout is in masterplan §16.
