The setup that gives the fastest edit loop is: run MySQL and Redis in Docker, and
run both apps directly on your machine. The gateway reloads on file change with
[`air`](https://github.com/air-verse/air); the frontend runs the Vite dev server,
which hot-reloads in the browser.

## Prerequisites

| Tool | Why | Notes |
|---|---|---|
| Go 1.26+ | Build and run the gateway | The Go toolchain auto-switches to the version pinned in `go.mod` (`GOTOOLCHAIN=auto`). |
| Node 22+ with pnpm | Build and run the frontend | Run `corepack enable` once to get pnpm. |
| Docker | Run MySQL and Redis | Used for local infrastructure only; both apps run on the host. |
| `air` | Gateway hot reload | Watches Go files and rebuilds on change. |
| `golangci-lint` | Gateway linter | Run with `make lint`. |

You do **not** need a C compiler. The gateway's WhatsApp keystore uses a pure-Go
SQLite driver, so everything builds with `CGO_ENABLED=0`.

## First run

Run these once, from the repo root, to set up config, start infrastructure, and
create the database tables:

```sh
cp deploy/.env.example deploy/.env    # gateway env — fill APP_ENCRYPTION_KEY, WHATSAPP_ADMIN_NUMBER, …
cp web/.env.example web/.env          # frontend env — fill BETTER_AUTH_SECRET, …
make infra-up                         # start MySQL + Redis (ports bound to localhost)
make migrate                          # create WA app-data tables (runs `server migrate up`)
cd web && pnpm install && pnpm db:migrate && cd ..   # create the auth tables (drizzle-kit)
```

Two values have no usable default and must be filled in before the apps start:

| Variable | File | What to set |
|---|---|---|
| `APP_ENCRYPTION_KEY` | `deploy/.env` | A base64-encoded 32-byte AES-GCM key. Generate with `openssl rand -base64 32`. Encrypts webhook secrets at rest. |
| `BETTER_AUTH_SECRET` | `web/.env` | The better-auth signing secret (32+ chars). Generate with `openssl rand -base64 32`. |

There are two separate migration tools, one per set of tables — don't mix them up:

- `make migrate` creates the WhatsApp app-data tables (gateways, sessions,
  contacts, …). The gateway binary owns these and embeds golang-migrate.
- `pnpm db:migrate` (run in `web/`) creates the better-auth tables (user,
  session, apikey, organization, …) with drizzle-kit.

## Day to day

After the first run, start the infrastructure once, then run each app in its own
terminal:

```sh
make infra-up    # MySQL + Redis (skip if already running)
make dev         # terminal 1: gateway under air; reloads on change
make web         # terminal 2: frontend dev server (pnpm dev) on :3000
```

The gateway listens on `:8080` by default. Open `http://localhost:3000` for the
frontend. The browser calls the gateway directly on `:8080`; the gateway's CORS
config already allows `http://localhost:3000` (via `FRONTEND_ORIGINS`), so the
two work together with no extra setup.

To wipe the dev database and start clean, run `make infra-reset` (this removes the
Docker volumes), then re-run the two migrate steps from the [first run](#first-run).

## Running the gateway in Docker instead

By default the gateway runs on your host under `air`. If you want it to run inside
a container instead — for example to match the production OS and Go toolchain
without installing Go locally — the dev compose file has two profiles. The
frontend still runs on your host under Vite in both cases.

| Command | What it does |
|---|---|
| `docker compose -f deploy/docker-compose.dev.yml --profile gateway up -d --build` | Runs the gateway from the production image (`deploy/Dockerfile`). Rebuild to pick up code changes. |
| `docker compose -f deploy/docker-compose.dev.yml --profile gateway-dev up -d --build` | Runs the gateway under `air` with the repo source mounted. Edit Go on the host; the container rebuilds. |

Both profiles publish `:8080`, so run only one at a time. Both reach the
host-side frontend's JWKS endpoint through `host.docker.internal` and apply the
WA migrations automatically on start. They read the same `deploy/.env` as the
host gateway; the compose file overrides only the values that must differ inside
a container (the MySQL and Redis hostnames, the JWKS host, and the keystore
path on a named volume).

## Environment variables

Gateway settings live in `deploy/.env` (template: `deploy/.env.example`).
Frontend settings live in `web/.env` (template: `web/.env.example`). Both
templates document every variable with its default. The ones a contributor
usually cares about first:

### Gateway (`deploy/.env`)

| Variable | Default | What it does |
|---|---|---|
| `HTTP_ADDR` | `:8080` | Address the gateway listens on. |
| `GATEWAY_ID` | `gw-1` | This gateway's id, stamped on the sessions it owns. |
| `PUBLIC_URL` | `http://localhost:8080` | External base URL clients use to reach this gateway. |
| `BETTER_AUTH_URL` | `http://localhost:3000` | The frontend's base URL. The gateway only trusts JWTs whose issuer and audience match it. No trailing slash. |
| `FRONTEND_ORIGINS` | `http://localhost:3000` | Comma-separated list of browser origins the gateway allows (CORS). |
| `APP_ENCRYPTION_KEY` | — | Base64 32-byte AES-GCM key; encrypts webhook secrets at rest. Required. |
| `MYSQL_DSN` | `gw:gwpass@tcp(127.0.0.1:3306)/gateway?…` | Shared MySQL. The gateway reads/writes the WA tables and reads the `apikey`/`user` tables. |
| `WHATSMEOW_STORE_DSN` | `file:store.db?…` | SQLite file holding the WhatsApp device keys. A local file in dev; the `/data/keystore` volume in Docker. |
| `REDIS_URL` | `redis://127.0.0.1:6379` | Redis for the work queue, rate limiting, idempotency, and event fan-out. |
| `PUBSUB_REDIS_URL` | falls back to `REDIS_URL` | Redis channel the gateway listens on for key/user revocations from the frontend. |
| `WHATSAPP_ADMIN_NUMBER` | — | Admin number the gateway pairs on boot. |

### Frontend (`web/.env`)

| Variable | Default | What it does |
|---|---|---|
| `BETTER_AUTH_SECRET` | — | better-auth signing/encryption secret (32+ chars). Required. |
| `BETTER_AUTH_URL` | `http://localhost:3000` | The frontend's own base URL, used as the JWT issuer and audience. |
| `DATABASE_URL` | `mysql://…@localhost:3306/wa_gateway` | MySQL connection for the frontend's Drizzle client: better-auth tables (read/write) plus read-only WA queries. |
| `GATEWAY_URL` | `http://localhost:8080` | Gateway base URL the frontend server calls. |
| `VITE_GATEWAY_URL` | `http://localhost:8080` | Same gateway URL exposed to the browser (Vite inlines `VITE_*` vars). |
| `PUBSUB_REDIS_URL` | `redis://localhost:6379` | Redis the frontend publishes revocations to (must match the gateway's `PUBSUB_REDIS_URL`). |
| `USER_REGISTRATION_ENABLED` | `true` | Whether people can sign themselves up. Set `false` to lock it down. |

<Callout type="warn">
The SQLite keystore (`WHATSMEOW_STORE_DSN`) holds each connected number's device
keys. If you lose that file, every number has to be paired again. It's a local
file in dev; in production keep it on a persistent named volume and back it up.
</Callout>

## Smoke test

`scripts/smoke.sh` runs the automatable part of the auth flow against a running
stack: it registers a user, gets a JWT from better-auth, calls the gateway with
that token, creates a session, and fetches the session's pairing QR code. It
stops with an error on any unexpected HTTP status. The steps that need a real
phone (pair, send, stream) are printed as manual instructions at the end.

```sh
BETTER_AUTH_URL=http://localhost:3000 GATEWAY_URL=http://localhost:8080 scripts/smoke.sh
```
