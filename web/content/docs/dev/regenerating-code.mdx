---
title: Regenerating code
description: The four generated files in this repo, the source each is built from, and the command to rebuild it.
---

Four files in this repo are generated from another file (or from the database).
You don't edit them by hand. When you change the source, run the matching command
to rebuild the generated file, then commit the result. Committing the output means
a fresh checkout builds without anyone having to run a generate step first.

All commands run from `web/`, unless a step says otherwise.

| Generated file | Built from | Command |
|---|---|---|
| `app/lib/api/schema.d.ts` — TypeScript types for the gateway client | `docs/openapi.yaml` (the gateway's API contract) | `pnpm gen:api` |
| `content/docs/api/**` — the API reference pages on the docs site | `docs/openapi.yaml` | `pnpm docs:openapi` |
| `app/lib/db/auth-schema.ts` — Drizzle table definitions for the auth tables | `app/lib/auth/server.ts` (the better-auth config) | `pnpm auth:generate` |
| `app/lib/db/wa.ts` — read-only Drizzle definitions for the WhatsApp tables | the live database, after gateway migrations are applied | `pnpm db:introspect` |

## `app/lib/api/schema.d.ts` — gateway client types

`pnpm gen:api` runs `openapi-typescript` over `docs/openapi.yaml` and writes the
TypeScript types that the frontend's gateway client and its React Query hooks are
built on. Run it whenever you edit `docs/openapi.yaml`.

## `content/docs/api/**` — API reference pages

`pnpm docs:openapi` rebuilds the whole `content/docs/api/**` tree from
`docs/openapi.yaml`. The script (`scripts/generate-openapi-docs.mts`) produces:

- one MDX page per API operation, grouped into folders by OpenAPI tag,
- a `meta.json` in each folder so the pages show up as an ordered tree in the
  docs sidebar, and
- a landing page (`index.mdx`) for the section.

The script deletes the `content/docs/api` directory before writing, so operations
you renamed or removed don't leave stale pages behind.

<Callout type="warn">
Do not hand-edit anything under `content/docs/api/`. The next `pnpm docs:openapi`
run deletes the directory and rewrites it, so your edits would be lost. Change
`docs/openapi.yaml` instead, then regenerate.
</Callout>

## `app/lib/db/auth-schema.ts` — auth table schema

`pnpm auth:generate` reads the better-auth config in `app/lib/auth/server.ts`
(including its plugins) and rewrites the Drizzle table definitions for the auth
tables. Run it after you change the better-auth plugin set or its options.

After regenerating, create and apply a migration with `pnpm db:migrate`
(drizzle-kit). `drizzle.config.ts` is scoped to the auth tables only, so check
that the generated migration touches no gateway-owned table — the gateway owns
the WhatsApp tables and is their only migration writer.

## `app/lib/db/wa.ts` — WhatsApp table read models

The frontend reads the WhatsApp app-data tables (gateways, sessions, contacts,
and so on) through the read-only Drizzle definitions in `app/lib/db/wa.ts`. The
gateway owns these tables and migrates them with golang-migrate; the frontend
never migrates them, it only mirrors them.

So you don't write `wa.ts` by hand against the Go schema. Instead, apply the
gateway migration to a database, then have drizzle-kit read that live database
and generate the definitions from it. This keeps `wa.ts` matching what the gateway
actually created.

After the WhatsApp schema changes:

```sh
make migrate            # apply the new gateway migration to your dev DB (from repo root)
cd web && pnpm db:introspect   # read the live schema into app/lib/db/wa.ts
```

`make migrate` runs `go run ./cmd/server migrate up`.
