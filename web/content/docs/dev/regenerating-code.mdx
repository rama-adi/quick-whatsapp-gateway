---
title: Regenerating code
description: Four generated artifacts, the command for each, and what to run them against.
---

Four things in the repo are generated from a source of truth. When the source
changes, regenerate the artifact and commit the result, so a fresh checkout
builds without a generate step. All commands run from `web/`.

| Artifact | Source of truth | Command | Output |
|---|---|---|---|
| Typed gateway client | `docs/openapi.yaml` | `pnpm gen:api` | `app/lib/api/schema.d.ts` |
| API reference docs | `docs/openapi.yaml` | `pnpm docs:openapi` | `content/docs/api/**` |
| better-auth table schema | `app/lib/auth/server.ts` | `pnpm auth:generate` | `app/lib/db/auth-schema.ts` |
| WA read-model mirror | the live gateway-migrated DB | `pnpm db:introspect` | `app/lib/db/wa.ts` |

## Typed gateway client — `pnpm gen:api`

Regenerates the TypeScript types the frontend's gateway client and React Query
hooks are built on, from the OpenAPI contract (`openapi-typescript`). Run it
whenever you change `docs/openapi.yaml`.

## API reference docs — `pnpm docs:openapi`

Regenerates the entire `content/docs/api/**` tree from the same OpenAPI contract
— one MDX page per operation, foldered by tag, with a generated landing page and
per-folder `meta.json`. The script cleans the output directory first, so renamed
or removed operations don't leave orphan pages. **Do not hand-edit anything under
`content/docs/api/`** — a regen overwrites it.

## better-auth schema — `pnpm auth:generate`

Reads the better-auth config (`app/lib/auth/server.ts`, including its plugins)
and regenerates the Drizzle table definitions for the auth plane. Run it after
changing the better-auth plugin set or options. After regenerating, produce and
review a migration with `pnpm db:migrate` — `drizzle.config.ts` is scoped to the
auth tables, so the diff must never touch a gateway-owned table.

## WA read-model mirror — `pnpm db:introspect`

The frontend reads the WA app-data tables through read-only Drizzle definitions
in `app/lib/db/wa.ts`. Those tables are **gateway-owned** (golang-migrate), so the
mirror is generated by introspecting a live database that has the gateway
migrations applied — never hand-maintained against the Go schema. After the WA
schema changes:

```sh
make migrate            # apply the new gateway migration to your dev DB (from repo root)
cd web && pnpm db:introspect   # pull the live schema into app/lib/db/wa.ts
```

This `introspect`-from-the-live-DB step is what keeps the read models from
drifting away from what the gateway actually wrote.
