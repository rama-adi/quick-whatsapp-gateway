This repo holds two codebases: the Go backend at the root and the web frontend
under `web/`. The Go backend runs as **two processes** — the router and the
gateway — so at runtime there are three services, sharing one MySQL database and
one Redis.

| Service | Location | Stack | What it does |
|---|---|---|---|
| **Router** | `cmd/router`, `internal/router` | Go | The public API service. It authenticates every caller, finds the gateway that owns the target session, forwards requests, and serves the realtime WebSocket and OpenAPI file. |
| **Gateway** | `cmd/server`, `internal/` | Go, whatsmeow | Holds the live WhatsApp connections and runs the REST handlers. It only accepts requests forwarded by the router. |
| **Frontend** | `web/` | TanStack Start, better-auth | The dashboard plus the identity service: login, registration, 2FA, organizations, and API-key management. Renders the dashboard, viewer, and contacts UI. |

The router and gateway are separate processes so you can run several gateways
behind one router later; today a single gateway is the norm. Splitting the
frontend off lets it run on serverless hosting while the Go backend runs close to
WhatsApp. None of the three need to be co-located.

## How the browser, frontend, router, and gateway connect

```
        Authorization: Bearer <JWT | api-key>
  Browser ──────────────────────────────────►  Router  (public API)
     │  ▲                                          │  • authenticates the caller (JWKS JWT / api-key)
     │  │ session cookie                           │  • resolves the gateway that owns the session
     │  │ mint JWT (/api/auth/token)               │  • mints a request-bound Ed25519 assertion
     ▼  │                                          ▼
  Frontend (TanStack Start + better-auth)       Gateway (Go, whatsmeow)  ──ws──► WhatsApp
     ├─ /api/auth/* (JWKS, token, orgs, keys)      │  • verifies the router's assertion (only)
     └─ read-only Drizzle reads ──┐                │  keystore → SQLite (gateway-local volume)
                                   ▼                ▼
                    MySQL  (auth tables ⇄ WA tables ⇄ gateways registry)
                                   ▲                │
   control bus: frontend PUBLISH ctrl:*  └── Redis ─┘  router SUBSCRIBES · work: queue / cache / fan-out
```

The browser talks to **one** base URL — the router — attaching a Bearer token.
The router authenticates the caller and proxies to the owning gateway; the
browser never addresses a gateway directly. The realtime feed is also on the
router (a WebSocket, see [Event shape](/docs/guides/event-shape)). The frontend
server does no gateway proxying. Its only server-side work is:

- the better-auth endpoints under `/api/auth/*`,
- minting short-lived gateway JWTs, and
- reading MySQL directly for server-side rendering.

## How a request is trusted

Authentication happens **once, on the router**. The gateway never sees an
end-user credential — it trusts only a token the router mints per request.

### 1. The router authenticates the caller

The frontend issues identity; the router verifies it. The router caches the JWKS
(the frontend's public signing keys) and reads the shared database directly, so
it never calls the frontend during a request. One middleware
(`internal/authz`) resolves both caller kinds, trying two acceptors in order:

| Caller | Credential | Router verifies by | Resolves to |
|---|---|---|---|
| **Human** (browser) | `Authorization: Bearer <JWT>` | JWKS signature + checking `iss` / `aud` / `exp`, all local and cached | user id, active org, org role, platform role |
| **Machine** (programmatic) | `Bearer <api-key>` or `x-api-key`, prefix `wa_` | SHA-256 hashing the key, then looking up the row in the shared `apikey` table | org id, key id, key permissions (no user) |

If neither acceptor matches, the request is rejected with `401`. The router then
finds the gateway that owns the target session, checks the caller's organization
owns it, and proxies on.

### 2. The gateway checks the router's proof

The router strips the end-user credential and attaches a request-bound
**internal assertion** in the `X-Internal-Assertion` header. This is a single-use
Ed25519-signed token (`internal/assertion`) that binds the caller's identity and
capabilities to *this exact request*: it carries the method, path, and a hash of
the body, an issuer (`iss`) and the target gateway's id (`aud`), a short expiry,
and a one-time nonce (`jti`).

The gateway holds only the router's **public** key — fetched from the router's
JWKS endpoint (`ROUTER_JWKS_URL`, default `/.well-known/router-jwks.json`) and
cached. Its `assertion.Middleware` verifies, in order: the signature, that `aud`
matches its own `GATEWAY_ID` and `iss` matches `ROUTER_ASSERTION_ISSUER`, the
expiry, that the method/path/body-hash match the request, and that the `jti` has
not been seen before (anti-replay). It then rebuilds the same `authz.Principal`
from the asserted claims and applies its capability gates and org-scoped queries
**exactly as before** — only the source of the principal changed, not the
downstream logic.

So the gateway does **no** JWKS or api-key verification anymore, and it mounts no
CORS — both moved to the router.

### Revoking access immediately

The short JWT TTL and a ~60 s cache TTL on validated keys bound how long stale
access can last on their own. To cut access off sooner, the frontend publishes on
a Redis **control bus** and the **router** subscribes:

| Channel | Router action |
|---|---|
| `ctrl:apikey.revoked` | Drop the cached key so the next request re-validates and fails closed. |
| `ctrl:user.banned` | Drop the user's cached keys and deny their in-flight JWTs until the JWT TTL ages out. |
| `ctrl:member.removed` | Deny the `(user, org)` pair until the JWT TTL ages out. |

Redis pub/sub is fire-and-forget: a router that is down when a message is
published misses it. Two backstops cover that gap — the ~60 s cache TTL (the
revoked row is gone, so the next lookup fails closed) and a one-time
reconciliation sweep on boot. Full detail: the
[trust model spec](https://github.com/ramaadi/quick-whatsapp-gateway/blob/main/docs/specs/trust-model.md)
and the [router spec](https://github.com/ramaadi/quick-whatsapp-gateway/blob/main/docs/specs/router.md).

## How the router finds the right gateway

Every session is pinned to the gateway that holds its WhatsApp keys, recorded in
`wa_sessions.gateway_id`. Gateways register themselves in a `gateways` table on
boot and heartbeat into it; the router reads that registry to route. A gateway
row moves through a lifecycle — `joining` → `active` → `draining` → `drained`,
plus `unreachable`, which the router derives when a heartbeat goes stale.

- **Routing an existing session:** the router looks up its `gateway_id` and
  proxies there. If that gateway is missing, not `active`, or stale, the request
  fails with `503 gateway_unavailable`.
- **Creating a session:** the router places it on the least-loaded `active`
  gateway.

Today there is one gateway with one self-row; running more than one is supported
by the schema but not the focus yet.

## Resources are owned by an organization, not a user

Every resource is owned by a better-auth `organization_id`, never by a user id.
Each user gets a personal organization on signup, so solo use is just a
one-member org, and "sharing a WhatsApp connection" means inviting someone into
the org. A user reaches a resource through org membership.

The router authorizes from the JWT's `activeOrganizationId` and `orgRole` claims
(or, for an api-key, its `reference_id`), carries them into the assertion, and the
gateway scopes every query to that org. Neither service reads the `member` table
while handling the request.

Capabilities map to roles and permissions as follows:

- **api-keys** carry explicit permission flags (`read` / `send` / `manage` /
  `events`); the flag is checked for the action.
- **JWT callers** map from org role: `owner` and `admin` get all four
  capabilities; `member` gets everything except `manage`.
- The platform `super_admin` role (from the JWT `role` claim) is allowed
  everything, across organizations.
