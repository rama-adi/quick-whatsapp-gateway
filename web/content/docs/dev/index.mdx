This repo is two services. They deploy independently and share one MySQL database.

| Service | Location | Stack | What it does |
|---|---|---|---|
| **Gateway** | repo root (`cmd/`, `internal/`, `migrations/`) | Go, whatsmeow | Holds the live WhatsApp connections. Exposes a JSON REST API, an NDJSON event stream, and outbound webhooks. Has no human login and serves no UI. |
| **Frontend** | `web/` | TanStack Start, better-auth | The dashboard plus the identity service: login, registration, 2FA, organizations, and API-key management. Renders the dashboard, viewer, and contacts UI. |

Splitting them lets the frontend run on serverless hosting while the gateway runs
close to WhatsApp. They do not need to be co-located.

The browser calls the gateway **directly** вАФ for both actions and the NDJSON
stream вАФ attaching a Bearer token. The frontend server never proxies gateway
traffic. Its only server-side work is:

- the better-auth endpoints under `/api/auth/*`,
- minting short-lived gateway JWTs, and
- reading MySQL directly for server-side rendering.

## How the gateway authenticates a request

The frontend issues identity; the gateway only verifies it. The gateway never
calls the frontend during a request вАФ it caches the JWKS (the frontend's public
signing keys) and reads the shared database directly.

There are exactly two kinds of caller. One middleware
(`internal/authz/middleware.go`) resolves both, trying two acceptors in order:

| Caller | Credential | Gateway verifies by | Resolves to |
|---|---|---|---|
| **Human** (browser) | `Authorization: Bearer <JWT>` | JWKS signature + checking `iss` / `aud` / `exp`, all local and cached | user id, active org, org role, platform role |
| **Machine** (programmatic) | `Bearer <api-key>` or `x-api-key`, prefix `wa_` | SHA-256 hashing the key, then looking up the row in the shared `apikey` table | org id, key id, key permissions (no user) |

If neither acceptor matches, the request is rejected with `401`. The resolved
identity is attached to the request context. Handlers then scope every query to
the caller's organization and gate the action by capability (`read` / `send` /
`manage` / `events`).

Two things a contributor will run into:

- **JWTs are short-lived (~5 min) EdDSA tokens.** A JWT (JSON Web Token) is a
  signed identity token. The browser holds the long-lived better-auth session
  cookie on the frontend's domain, calls `/api/auth/token` to mint a JWT from it,
  and attaches that JWT to gateway calls. There is no separate refresh token: the
  session is the revocable credential, and the JWT is a short access token minted
  from it.
- **The api-key path re-implements better-auth's hash.** The `apikey.key` column
  stores `base64url(SHA-256(rawKey))`, unpadded вАФ not hex, not padded base64.
  The `reference_id` column holds the owning org id. This hash and column layout
  are pinned by contract tests in `internal/authz/` that mint a real better-auth
  credential and verify it in the gateway. Bumping better-auth to a new major
  version means re-running those tests.

### Revoking access immediately

The short JWT TTL and a ~60 s cache TTL on validated keys bound how long stale
access can last on their own. To cut access off sooner, the frontend publishes
on a Redis **control bus** and every gateway subscribes:

| Channel | Gateway action |
|---|---|
| `ctrl:apikey.revoked` | Drop the cached key and any live streams it authenticated. |
| `ctrl:user.banned` | Drop the user's cached keys, drop all their live streams, and deny-list their in-flight JWTs until the JWT TTL ages out. |
| `ctrl:member.removed` | Deny the `(user, org)` pair until the JWT TTL ages out and drop that user's streams for the org. |

Redis pub/sub is fire-and-forget: a gateway that is down when a message is
published misses it. Two backstops cover that gap вАФ the ~60 s cache TTL (the
revoked row is gone, so the next lookup fails closed) and a one-time
reconciliation sweep on boot. Full detail: the
[trust model spec](https://github.com/ramaadi/quick-whatsapp-gateway/blob/main/docs/specs/trust-model.md).

## Resources are owned by an organization, not a user

Every resource is owned by a better-auth `organization_id`, never by a user id.
Each user gets a personal organization on signup, so solo use is just a
one-member org, and "sharing a WhatsApp connection" means inviting someone into
the org. A user reaches a resource through org membership.

The gateway authorizes from the JWT's `activeOrganizationId` and `orgRole`
claims (or, for an api-key, its `reference_id`). It does not join the `member`
table during a request.

Capabilities map to roles and permissions as follows:

- **api-keys** carry explicit permission flags (`read` / `send` / `manage` /
  `events`); the gateway checks the flag for the action.
- **JWT callers** map from org role: `owner` and `admin` get all four
  capabilities; `member` gets everything except `manage`.
- The platform `super_admin` role (from the JWT `role` claim) is allowed
  everything, across organizations.
