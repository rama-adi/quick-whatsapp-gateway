There are two services. They share one MySQL database and one Redis, and the
gateway keeps its WhatsApp keystore (SQLite) on a volume.

| Service | Image | Port | Owns |
|---|---|---|---|
| Gateway | `deploy/Dockerfile` (Go, ~52 MB, no CGO) | `8080` | WhatsApp sessions; writes the WA tables; the SQLite keystore |
| Frontend | `deploy/Dockerfile.web` (Node 22, TanStack Start) | `3000` | better-auth; writes the auth tables; reads the WA tables for display |
| MySQL | `mysql:8.4` | `3306` | both table sets in one database |
| Redis | `redis:7-alpine` | `6379` | the work bus and the control bus |

## Run it with Docker Compose

`deploy/docker-compose.yml` runs all four. Put the three secrets in
`deploy/.env`, then build and start:

```bash
cd deploy
cat > .env <<'EOF'
APP_ENCRYPTION_KEY=...   # openssl rand -base64 32 — encrypts webhook secrets at rest
BETTER_AUTH_SECRET=...   # openssl rand -base64 32 — signs sessions, encrypts the JWKS private key
WHATSAPP_ADMIN_NUMBER=   # optional — the number paired on boot; leave empty to skip
EOF
docker compose up --build -d
```

The browser uses `http://localhost:3000` (frontend) and `http://localhost:8080`
(gateway). Inside the network the frontend calls the gateway at `gateway:8080`
and the gateway fetches the frontend's JWKS at `frontend:3000/api/auth/jwks`.

## Migrations

The two table sets are created by different tools, and only one runs on its own.

- **WA tables (gateway):** created and upgraded automatically. The gateway runs
  its embedded migrations on every boot. Nothing to run.
- **Auth tables (frontend):** not automatic. The runtime frontend image has no
  `drizzle-kit`, so apply them once against the database before first use:

  ```bash
  cd web
  DATABASE_URL='mysql://gw:gwpass@<db-host>:3306/gateway' pnpm db:migrate
  ```

<Callout type="warn">
A fresh stack boots with the WA tables present but the auth tables missing.
Sign-up and login return 500 until you run `pnpm db:migrate`. Run it as the
first step of a deploy, or add it to your release pipeline.
</Callout>

## Volumes and backups

| Volume | Mount | Holds | If lost |
|---|---|---|---|
| `keystore_data` | gateway `/data/keystore` | WhatsApp device crypto (SQLite) | **every number must be re-paired** |
| `mysql_data` | MySQL `/var/lib/mysql` | all WA + auth data | full data loss |
| `redis_data` | Redis `/data` | queue and cache | nothing — it rebuilds |

Back up `keystore_data` like a database. It holds the device keys, so losing it
means re-pairing every connected number, and it is what ties a session to a
specific gateway.

## Configuration

Every variable is listed in `deploy/.env.example` (gateway) and
`web/.env.example` (frontend). These few must agree across the two services:

| Variable | Set on | Value |
|---|---|---|
| `BETTER_AUTH_URL` | both | the frontend's public URL — the JWT `iss`/`aud` the gateway enforces |
| `FRONTEND_ORIGINS` | gateway | the browser origin(s) allowed to call the gateway (CORS) |
| `BETTER_AUTH_JWKS_URL` | gateway | where the gateway fetches JWKS — the *internal* frontend URL when it differs from the public one (e.g. `http://frontend:3000/api/auth/jwks`) |
| `GATEWAY_URL` / `VITE_GATEWAY_URL` | frontend | the gateway URL for server code / for the browser |
| `PUBSUB_REDIS_URL` | both | the shared control-bus Redis — the frontend publishes key/user revocations, the gateway subscribes |

<Callout type="info">
`BETTER_AUTH_URL` and `BETTER_AUTH_JWKS_URL` can hold different hosts. The first
is the public URL the token's `iss`/`aud` must match; the second is only where
the gateway fetches the keys. When the gateway reaches the frontend by an
internal name, set `BETTER_AUTH_JWKS_URL` to that name and leave
`BETTER_AUTH_URL` as the public URL.
</Callout>

## Create the first admin

- **Platform super-admin:** put the better-auth user id(s) in `ADMIN_USER_IDS`
  on the frontend. A user signs up normally, then you add their id.
- **Admin WhatsApp number:** set `WHATSAPP_ADMIN_NUMBER` on the gateway. The
  gateway pairs it on boot and prints the pairing code to its logs and the admin
  panel.

## Hosting the gateway and frontend separately

The frontend can run anywhere it can reach MySQL and the gateway; the gateway
runs near WhatsApp with its keystore volume. The gateway verifies callers
against the JWKS and the shared `apikey` table, so it never calls back to the
frontend on a request.

- **Frontend on a separate host or serverless:** point `DATABASE_URL` at the
  shared MySQL (use a pooler on serverless) and `GATEWAY_URL`/`VITE_GATEWAY_URL`
  at the gateway. The browser talks to the gateway directly.
- **Your own MySQL/Redis:** `deploy/docker-compose.external.yml` runs the two
  app images against services you provide.
- **JWKS host:** if the gateway fetches JWKS from a host other than
  `BETTER_AUTH_URL`, add that host to the frontend's `TRUSTED_ORIGINS`.

## Health checks

The gateway serves `GET /healthz` (liveness), `GET /readyz` (checks MySQL and
Redis), and `GET /metrics` (Prometheus). Use `/readyz` for the container
healthcheck.

## Running the gateway in Docker for development

Day to day you run both apps on the host (see
[Local development](/docs/dev/local-development)). To run the gateway in a
container instead — same OS and toolchain as production, with the frontend still
on the host under Vite — use one of two compose profiles:

```bash
# Hot reload: air watches the mounted source and rebuilds in the container.
docker compose -f deploy/docker-compose.dev.yml --profile gateway-dev up -d --build

# Or the production image (rebuild to pick up code changes):
docker compose -f deploy/docker-compose.dev.yml --profile gateway up -d --build
```

Both reach the host frontend's JWKS through `host.docker.internal` (already
allowed in `vite.config.ts`) and apply the WA migrations on boot. Run only one
at a time — both bind port 8080.

To add or override gateway env vars for these profiles, create
`deploy/gateway.dev.env` (one `KEY=value` per line; gitignored). Values already
set in the compose `environment:` block — `MYSQL_DSN`, `REDIS_URL`,
`BETTER_AUTH_URL`, and the rest — take precedence over the file, so change those
by editing the compose block. See `deploy/gateway.dev.env.example`.

## Pairing a WhatsApp number

Pairing is the one step a deploy can't script. Create a session (or set
`WHATSAPP_ADMIN_NUMBER`), open the QR or pairing code in the dashboard, and link
it from the phone's WhatsApp app. This uses an unofficial client — automated use
may violate WhatsApp's terms and risks a ban.
