The system is two services. They share one MySQL database and one Redis. The
gateway also keeps its WhatsApp keystore in a SQLite file on a volume.

| Service | Image | Port | Owns |
|---|---|---|---|
| Gateway | `deploy/Dockerfile` (Go, ~52 MB, no CGO) | `8080` | WhatsApp sessions; writes the WA tables; the SQLite keystore |
| Frontend | `deploy/Dockerfile.web` (Node 22, TanStack Start) | `3000` | better-auth; writes the auth tables; reads the WA tables for display |
| MySQL | `mysql:8.4` | `3306` | both table sets in one database |
| Redis | `redis:7-alpine` | `6379` | the work queue and the control bus |

## Run it with Docker Compose

`deploy/docker-compose.yml` runs all four services. Put the three secrets in
`deploy/.env`, then build and start:

```bash
cd deploy
cat > .env <<'EOF'
APP_ENCRYPTION_KEY=...   # openssl rand -base64 32 — encrypts webhook secrets at rest
BETTER_AUTH_SECRET=...   # openssl rand -base64 32 — signs sessions, encrypts the JWKS private key
WHATSAPP_ADMIN_NUMBER=   # optional — the number paired on boot; leave empty to skip
EOF
docker compose up --build -d
```

The browser uses `http://localhost:3000` (frontend) and `http://localhost:8080`
(gateway). Inside the Docker network the frontend calls the gateway at
`gateway:8080`, and the gateway fetches the frontend's signing keys at
`frontend:3000/api/auth/jwks`.

## Migrations

There are two sets of tables, created by two different tools. Only one runs on
its own.

| Tables | Created by | When |
|---|---|---|
| WA tables (gateway) | gateway's embedded migrations | automatically, on every boot — nothing to run |
| Auth tables (frontend) | `pnpm db:migrate` (drizzle-kit) | you run it once, before first use |

The runtime frontend image does not include `drizzle-kit`, so apply the auth
tables against the database yourself:

```bash
cd web
DATABASE_URL='mysql://gw:gwpass@<db-host>:3306/gateway' pnpm db:migrate
```

<Callout type="warn">
A fresh stack boots with the WA tables present but the auth tables missing.
Sign-up and login return 500 until you run `pnpm db:migrate`. Run it as the
first step of a deploy, or add it to your release pipeline.
</Callout>

## Volumes and backups

| Volume | Mount | Holds | If lost |
|---|---|---|---|
| `keystore_data` | gateway `/data/keystore` | WhatsApp device keys (SQLite) | **every number must be re-paired** |
| `mysql_data` | MySQL `/var/lib/mysql` | all WA + auth data | full data loss |
| `redis_data` | Redis `/data` | queue and cache | nothing — it rebuilds |

<Callout type="warn">
Back up `keystore_data` like a database. It holds the WhatsApp device keys, so
losing it means re-pairing every connected number. It is also what ties a
session to a specific gateway. Always keep it on a named volume in production.
</Callout>

## Configuration

Every gateway variable is listed in `deploy/.env.example`; every frontend
variable in `web/.env.example`. The full tables are [below](#environment-variable-reference).

A few variables must hold matching values across the two services. Get these
right first:

| Variable | Set on | Value |
|---|---|---|
| `BETTER_AUTH_URL` | both | the frontend's public URL — the JWT `iss`/`aud` the gateway enforces |
| `FRONTEND_ORIGINS` | gateway | the browser origin(s) allowed to call the gateway (CORS) |
| `BETTER_AUTH_JWKS_URL` | gateway | where the gateway fetches the signing keys — the *internal* frontend URL when it differs from the public one (e.g. `http://frontend:3000/api/auth/jwks`) |
| `GATEWAY_URL` / `VITE_GATEWAY_URL` | frontend | the gateway URL for server code / for the browser |
| `PUBSUB_REDIS_URL` | both | the shared control-bus Redis — the frontend publishes key/user revocations, the gateway subscribes |

A JWT (JSON Web Token) is the signed login token the frontend issues to the
browser. JWKS (JSON Web Key Set) is the public-key document the gateway fetches
to verify those tokens. `iss`/`aud` are the issuer and audience fields inside
the token; the gateway only accepts tokens whose `iss`/`aud` equal
`BETTER_AUTH_URL`.

<Callout type="info">
`BETTER_AUTH_URL` and `BETTER_AUTH_JWKS_URL` can hold different hosts. The first
is the public URL the token's `iss`/`aud` must match; the second is only where
the gateway fetches the keys. When the gateway reaches the frontend by an
internal name, set `BETTER_AUTH_JWKS_URL` to that name and leave
`BETTER_AUTH_URL` as the public URL.
</Callout>

### Environment-variable reference

**Gateway** (`deploy/.env.example`):

| Variable | Default | What it does |
|---|---|---|
| `HTTP_ADDR` | `:8080` | Address the gateway listens on. |
| `GATEWAY_ID` | `gw-1` | This gateway's id, stamped on the sessions it holds. |
| `PUBLIC_URL` | — | External base URL clients use to reach this gateway. |
| `BETTER_AUTH_URL` | — | The frontend's base URL. The gateway only trusts tokens issued by it. |
| `BETTER_AUTH_JWKS_URL` | `${BETTER_AUTH_URL}/api/auth/jwks` | Where the gateway fetches the signing keys. Set it when the internal frontend URL differs from the public one. |
| `FRONTEND_ORIGINS` | — | Comma-separated browser origins allowed to call the gateway (CORS). |
| `APP_ENCRYPTION_KEY` | — | Base64 32-byte AES-GCM key (`openssl rand -base64 32`). Encrypts webhook secrets at rest. |
| `MYSQL_DSN` | — | Shared database. The gateway writes WhatsApp data and reads auth data. |
| `WHATSMEOW_STORE_DSN` | `file:store.db?...` | SQLite file holding the WhatsApp device keys. In containers, point it at `/data/keystore/store.db` on a persistent volume. |
| `REDIS_URL` | — | Redis for the send queue, rate limiting, idempotency, and event fan-out. |
| `PUBSUB_REDIS_URL` | falls back to `REDIS_URL` | Control-bus Redis for cross-service key/user revocation. Same as `REDIS_URL` on a single-instance deploy. |
| `REDIS_PREFIX` | `gw` | Key/channel namespace, so several stacks can share one Redis. |
| `WHATSAPP_ADMIN_NUMBER` | — | A number the gateway pairs automatically on boot. See [the admin number](#the-admin-number). |
| `WHATSAPP_ADMIN_CMD_PREFIX` | `am` | Command prefix on the admin session's private channel. See [the admin number](#the-admin-number). |
| `DEFAULT_RATE_PER_MIN` / `DEFAULT_RATE_PER_HOUR` | `20` / `200` | Default send limits per session. |
| `DEFAULT_AUTO_READ` | `true` | Mark incoming messages read before replying. |

<Callout type="warn">
The SQLite keystore at `WHATSMEOW_STORE_DSN` holds the device crypto material.
Lose that file and every number has to be re-paired. In production, mount it on
a named volume and back it up.
</Callout>

**Frontend** (`web/.env.example`):

| Variable | What it does |
|---|---|
| `BETTER_AUTH_SECRET` | Secret used to sign sessions and tokens. |
| `BETTER_AUTH_URL` | The frontend's own base URL. |
| `DATABASE_URL` | The shared database (the same one the gateway uses). |
| `GATEWAY_URL` | Base URL the frontend's server code calls for actions and the event stream. |
| `VITE_GATEWAY_URL` | Same gateway URL, but exposed to the browser. |
| `PUBSUB_REDIS_URL` | Control-bus Redis the frontend publishes revocations to. |
| `USER_REGISTRATION_ENABLED` | Whether people can sign themselves up. See [Lock down self-registration](#lock-down-self-registration). |
| `ADMIN_USER_IDS` | Comma-separated better-auth user ids granted platform super-admin. See [Create the first admin](#create-the-first-admin). |

### Lock down self-registration

Self-registration is on by default. To lock it down — so only an admin can add
people — set `USER_REGISTRATION_ENABLED=false` on the frontend and create users
from the admin surface instead.

## Create the first admin

There are two separate kinds of "admin", set independently:

- **Platform super-admin:** put the better-auth user id(s) in `ADMIN_USER_IDS`
  on the frontend. A user signs up normally, then you add their id.
- **Admin WhatsApp number:** set `WHATSAPP_ADMIN_NUMBER` on the gateway. See
  below.

### The admin number

A deployment can name one number as its admin number through the gateway's
`WHATSAPP_ADMIN_NUMBER` setting. On boot, if that number is not already linked,
the gateway creates its session, prints a pairing code to its logs, and shows
the same code on the admin panel — so you can pair it without a browser.

The admin number is otherwise an ordinary session: it sends and receives like
any other. It also reserves a private command channel. Messages it receives that
start with the admin command prefix (`WHATSAPP_ADMIN_CMD_PREFIX`, default `am`)
are handled internally — they are **not** stored, emitted as events, or counted
against rate limits. In this release that channel is reserved but does nothing;
commands arrive in a later version.

## Hosting the gateway and frontend separately

The frontend can run anywhere it can reach MySQL and the gateway. The gateway
runs near WhatsApp with its keystore volume. The gateway verifies callers
against the signing keys (JWKS) and the shared `apikey` table, so it never calls
back to the frontend on a request.

| Goal | What to do |
|---|---|
| Run the frontend on a separate host or serverless | Point `DATABASE_URL` at the shared MySQL (use a connection pooler on serverless) and `GATEWAY_URL`/`VITE_GATEWAY_URL` at the gateway. The browser talks to the gateway directly. |
| Bring your own MySQL/Redis | `deploy/docker-compose.external.yml` runs the two app images against services you provide. |
| Gateway fetches signing keys from a different host | If the gateway fetches JWKS from a host other than `BETTER_AUTH_URL`, add that host to the frontend's `TRUSTED_ORIGINS`. |

## Health checks

The gateway serves three endpoints:

| Endpoint | Purpose |
|---|---|
| `GET /healthz` | Liveness. |
| `GET /readyz` | Readiness — checks MySQL and Redis. Use this for the container healthcheck. |
| `GET /metrics` | Prometheus metrics. |

## Running the gateway in Docker for development

Day to day you run both apps on the host (see
[Local development](/docs/dev/local-development)). To run the gateway in a
container instead — same OS and toolchain as production, with the frontend still
on the host under Vite — use one of two compose profiles:

```bash
# Hot reload: air watches the mounted source and rebuilds in the container.
docker compose -f deploy/docker-compose.dev.yml --profile gateway-dev up -d --build

# Or the production image (rebuild to pick up code changes):
docker compose -f deploy/docker-compose.dev.yml --profile gateway up -d --build
```

Both profiles reach the host frontend's signing keys through
`host.docker.internal` (already allowed in `vite.config.ts`) and apply the WA
migrations on boot. Run only one at a time — both bind port 8080.

Both profiles read the **same** gateway env file the host gateway uses,
`deploy/.env` (copy it from `deploy/.env.example`), so there is one place to set
gateway config. The compose `environment:` block overrides only the variables
that must differ inside a container: `MYSQL_DSN` and `REDIS_URL` (the service
hostnames), `BETTER_AUTH_JWKS_URL` (the frontend is on the host), and
`WHATSMEOW_STORE_DSN` (a named volume). Change anything else in `deploy/.env`.

## Pairing a WhatsApp number

Pairing is the one step a deploy cannot script. Create a session (or set
`WHATSAPP_ADMIN_NUMBER`), open the QR or pairing code in the dashboard, and link
it from the phone's WhatsApp app. This uses an unofficial client — automated use
may violate WhatsApp's terms and risks a ban.
