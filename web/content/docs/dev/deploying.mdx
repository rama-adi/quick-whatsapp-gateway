There are two supported deployment stories:

| Mode | Use it when | Deploy shape |
|---|---|---|
| **Self-host single box** | You want one private WhatsApp gateway for yourself, a team, or a small internal app. | One server runs one app container (router + gateway + frontend), MySQL, Redis, and the gateway keystore volume. |
| **Split production** | You are operating this as a product or a larger platform. | The frontend, router API server, database, Redis, and one or more gateways are hosted and scaled separately. |

Both modes use the same runtime pieces. The difference is packaging and who owns
the operational work. The self-host path stitches the router, gateway, and
frontend into one app image, includes MySQL and Redis in Compose, and defaults
self-registration off. The scaler path gives you the foundations: a modular
frontend/identity service, a router API server, gateway processes that register
and heartbeat into MySQL, organization-scoped data ownership, better-auth
api-keys, and role/capability checks. You still own capacity automation, billing,
tenant onboarding policy, and any ACL finer than the built-in org roles and
api-key permissions.

The system runs as three app services — the router, the gateway, and the
frontend — plus one MySQL database and one Redis they share. The gateway also
keeps its WhatsApp keystore in a SQLite file on a volume.

| Piece | Image | Port | Owns |
|---|---|---|---|
| Self-host app | `deploy/Dockerfile.selfhost` (Node 22 + two Go binaries) | `3000`, `8090` | Runs the router, gateway, and frontend in one container. Gateway port `8080` is localhost-only inside the container. |
| Router | `deploy/Dockerfile.router` (Go, no CGO) | `8090` | Public API service: authenticates callers, routes to the owning gateway, serves the realtime WebSocket and the OpenAPI contract |
| Gateway | `deploy/Dockerfile` (Go, no CGO) | `8080` (internal) | WhatsApp sessions; writes the WA tables; stores the SQLite keystore |
| Frontend | `deploy/Dockerfile.web` (Node 22, TanStack Start) | `3000` | better-auth and dashboard; writes auth tables; reads WA tables for display |
| MySQL | `mysql:8.4` | `3306` | both table sets in one database |
| Redis | `redis:7-alpine` | `6379` | the work bus and the control bus |

The **router** is the only API service clients reach. Publish the router; keep
the gateway internal.

## Self-host single box

Use `deploy/docker-compose.selfhost.yml` when you want one host to run
everything. It builds one app container with the router, gateway, and frontend
together. Compose still runs MySQL and Redis as separate services so their data
has normal volumes and backup boundaries. The app publishes the dashboard on
`3000` and the router API on `8090`; the gateway listens only on localhost
inside the app container.

Put the secrets in `deploy/.env`, then build and start:

```bash
cd deploy
cat > .env <<'EOF'
ROUTER_ED25519_PRIVATE_KEY=...   # openssl rand 32 | base64 — signs the router's internal assertions
APP_ENCRYPTION_KEY=...           # openssl rand -base64 32 — encrypts webhook secrets at rest
BETTER_AUTH_SECRET=...           # openssl rand -base64 32 — signs sessions, encrypts the JWKS private key
USER_REGISTRATION_ENABLED=false  # recommended for private self-hosts
WHATSAPP_ADMIN_NUMBER=           # optional — the number paired on boot; leave empty to skip
EOF
docker compose -f docker-compose.selfhost.yml up --build -d
```

The browser uses `http://localhost:3000` (frontend) and `http://localhost:8090`
(router); it never calls the gateway directly. Inside the app container: the
frontend calls the router at `127.0.0.1:8090`, the router fetches the
frontend's signing keys at `127.0.0.1:3000/api/auth/jwks` and proxies to the
gateway at `127.0.0.1:8080`, and the gateway fetches the router's keys at
`127.0.0.1:8090/.well-known/router-jwks.json` to verify the router's assertions.

<Callout type="info">
The self-host app container is not a single process. Its entrypoint supervises
three processes: `router`, `gateway`, and the TanStack Start Node server. If any
one exits, the container exits so Compose can restart the whole app.
</Callout>

### Lock down self-registration

Self-registration is off by default in `docker-compose.selfhost.yml`. Keep
`USER_REGISTRATION_ENABLED=false` if this is a private gateway and you do not
want random people creating accounts or pairing WhatsApp numbers. Turn it on
only when open sign-up is part of your operating model.

After the first trusted user signs up, put their better-auth user id in
`ADMIN_USER_IDS` and restart the frontend to grant platform super-admin rights.

## Split production topology

Use the split topology when this is a platform rather than one private gateway.
In this mode:

- The **frontend** can run separately, including serverless hosting, as long as
  it can reach MySQL and publish control-bus messages to Redis.
- The **router** is the public API service. It verifies better-auth
  JWTs and api-keys from shared MySQL, resolves session ownership, and proxies
  to gateways with request-bound Ed25519 assertions.
- **Gateways** run close to their WhatsApp connections. Each gateway has a stable
  `GATEWAY_ID`, a persistent SQLite keystore volume, and a `PUBLIC_URL` reachable
  by the router. Gateways register and heartbeat through the `gateways` table.
- New session creation is placed on the least-loaded active gateway. Existing
  sessions are pinned by `wa_sessions.gateway_id`; if the owning gateway is not
  active or has a stale heartbeat, the router returns `503 gateway_unavailable`.

`deploy/docker-compose.external.yml` shows the split-hosting shape for app
containers over your own MySQL and Redis:

```bash
docker compose -f deploy/docker-compose.external.yml up -d
```

For a SaaS product, this repo does not decide your capacity automation, tenant
onboarding, billing, quotas, extra gateway operations, or access rules beyond
organization roles and api-key permissions (`read`, `send`, `manage`, `events`).

## Migrations

There are two sets of tables, created by two different tools. Only one runs on
its own.

| Tables | Created by | When |
|---|---|---|
| WA tables (gateway) | gateway's embedded migrations | automatically, on every boot — nothing to run |
| Auth tables (frontend) | `pnpm db:migrate` (drizzle-kit) | you run it once, before first use |

The runtime frontend image does not include `drizzle-kit`, so apply the auth
tables against the database yourself:

```bash
cd web
DATABASE_URL='mysql://gw:gwpass@<db-host>:3306/gateway' pnpm db:migrate
```

<Callout type="warn">
A fresh stack boots with the WA tables present but the auth tables missing.
Sign-up and login return 500 until you run `pnpm db:migrate`. Run it as the
first step of a deploy, or add it to your release pipeline.
</Callout>

## Volumes and backups

| Volume | Mount | Holds | If lost |
|---|---|---|---|
| `keystore_data` | gateway `/data/keystore` | WhatsApp device keys (SQLite) | **every number must be re-paired** |
| `mysql_data` | MySQL `/var/lib/mysql` | all WA + auth data | full data loss |
| `redis_data` | Redis `/data` | queue and cache | nothing — it rebuilds |

<Callout type="warn">
Back up `keystore_data` like a database. It holds the WhatsApp device keys, so
losing it means re-pairing every connected number. It is also what ties a
session to a specific gateway. Always keep it on a named volume in production.
</Callout>

## Configuration

Every gateway variable is listed in `deploy/.env.example`; every frontend
variable in `web/.env.example`. The full tables are [below](#environment-variable-reference).

A few variables must hold matching values across the services. Get these right
first:

| Variable | Set on | Value |
|---|---|---|
| `ROUTER_ED25519_PRIVATE_KEY` | router | the Ed25519 key the router signs internal assertions with (`openssl rand 32 | base64`). Required — the router won't start without it. |
| `ROUTER_ISSUER` / `ROUTER_ASSERTION_ISSUER` | router / gateway | the assertion `iss` the router stamps and the gateway requires — these two must be equal |
| `ROUTER_JWKS_URL` | gateway | where the gateway fetches the router's public keys to verify assertions (the *internal* router URL, e.g. `http://router:8090/.well-known/router-jwks.json`) |
| `BETTER_AUTH_URL` | router + frontend | the frontend's public URL — the JWT `iss`/`aud` the router enforces |
| `BETTER_AUTH_JWKS_URL` | router | where the router fetches the frontend's signing keys — the *internal* frontend URL when it differs from the public one (e.g. `http://frontend:3000/api/auth/jwks`) |
| `FRONTEND_ORIGINS` | router | the browser origin(s) allowed to call the router (CORS) |
| `GATEWAY_URL` / `VITE_GATEWAY_URL` | frontend | the **router** URL for server code / for the browser |
| `PUBSUB_REDIS_URL` | router + frontend | the shared control-bus Redis — the frontend publishes key/user revocations, the router subscribes; defaults to `REDIS_URL` on the router |

A JWT (JSON Web Token) is the signed login token the frontend issues to the
browser. JWKS (JSON Web Key Set) is the public-key document the router fetches to
verify those tokens. `iss`/`aud` are the issuer and audience fields inside the
token; the router only accepts tokens whose `iss`/`aud` equal `BETTER_AUTH_URL`.
The router→gateway assertion uses a *second* JWKS — the router's own keys, which
the gateway fetches from `ROUTER_JWKS_URL`.

<Callout type="info">
`BETTER_AUTH_URL` and `BETTER_AUTH_JWKS_URL` can hold different hosts. The first
is the public URL the token's `iss`/`aud` must match; the second is only where
the router fetches the keys. When the router reaches the frontend by an internal
name, set `BETTER_AUTH_JWKS_URL` to that name and leave `BETTER_AUTH_URL` as the
public URL. The same split applies to the gateway's `ROUTER_JWKS_URL`.
</Callout>

### Environment-variable reference

`deploy/.env.example` is one file read by both Go processes. The first block is
the router's; the rest is the gateway's.

**Router** (`deploy/.env.example`):

| Variable | Default | What it does |
|---|---|---|
| `ROUTER_HTTP_ADDR` | `:8090` | Address the router listens on. |
| `ROUTER_PUBLIC_URL` | — | The router's external base URL — what clients use; the realtime WebSocket URL is derived from it. |
| `ROUTER_ISSUER` | `router` | The `iss` the router stamps on its internal assertions. Must match the gateway's `ROUTER_ASSERTION_ISSUER`. |
| `ROUTER_ED25519_PRIVATE_KEY` | — | Ed25519 key for signing internal assertions (`openssl rand 32 | base64`). Required. |
| `BETTER_AUTH_URL` | — | The frontend's base URL. The router only trusts JWTs issued by it. |
| `BETTER_AUTH_JWKS_URL` | `${BETTER_AUTH_URL}/api/auth/jwks` | Where the router fetches the frontend's signing keys. Set it when the internal frontend URL differs from the public one. |
| `FRONTEND_ORIGINS` | — | Comma-separated browser origins allowed to call the router (CORS). |
| `PUBSUB_REDIS_URL` | falls back to `REDIS_URL` | Control-bus Redis the router subscribes to for key/user/member revocations. |

**Gateway** (`deploy/.env.example`):

| Variable | Default | What it does |
|---|---|---|
| `HTTP_ADDR` | `:8080` | Address the gateway listens on (internal). |
| `GATEWAY_ID` | `gw-1` | This gateway's id, stamped on the sessions it holds; also the assertion `aud` it requires. |
| `PUBLIC_URL` | — | The gateway's internal base URL — the address the router proxies to (not a client-facing URL). |
| `ROUTER_JWKS_URL` | — | The router's JWKS, used to verify internal assertions. Required for the gateway. |
| `ROUTER_ASSERTION_ISSUER` | `router` | The `iss` the gateway requires on assertions; must match the router's `ROUTER_ISSUER`. |
| `APP_ENCRYPTION_KEY` | — | Base64 32-byte AES-GCM key (`openssl rand -base64 32`). Encrypts webhook secrets at rest. |
| `MYSQL_DSN` | — | Shared database. The gateway writes WhatsApp data and reads auth data. |
| `WHATSMEOW_STORE_DSN` | `file:store.db?...` | SQLite file holding the WhatsApp device keys. In containers, point it at `/data/keystore/store.db` on a persistent volume. |
| `REDIS_URL` | — | Work Redis for the send queue, rate limiting, idempotency, and event fan-out. |
| `REDIS_PREFIX` | `gw` | Key/channel namespace, so several stacks can share one Redis. |
| `WHATSAPP_ADMIN_NUMBER` | — | A number the gateway pairs automatically on boot. See [the admin number](#the-admin-number). |
| `WHATSAPP_ADMIN_ORG_ID` | — | The organization that owns the admin session. Required when `WHATSAPP_ADMIN_NUMBER` is set. |
| `WHATSAPP_ADMIN_CMD_PREFIX` | `am` | Command prefix on the admin session's private channel. See [the admin number](#the-admin-number). |
| `DEFAULT_RATE_PER_MIN` / `DEFAULT_RATE_PER_HOUR` | `20` / `200` | Default send limits per session. |
| `DEFAULT_AUTO_READ` | `true` | Mark incoming messages read before replying. |

<Callout type="warn">
The SQLite keystore at `WHATSMEOW_STORE_DSN` holds the device crypto material.
Lose that file and every number has to be re-paired. In production, mount it on
a named volume and back it up.
</Callout>

**Frontend** (`web/.env.example`):

| Variable | What it does |
|---|---|
| `BETTER_AUTH_SECRET` | Secret used to sign sessions and tokens. |
| `BETTER_AUTH_URL` | The frontend's own base URL. |
| `DATABASE_URL` | The shared database (the same one the gateway uses). |
| `GATEWAY_URL` | The **router** base URL the frontend's server code calls for actions and server functions. |
| `VITE_GATEWAY_URL` | The same **router** URL, exposed to the browser. The browser calls the router, not the gateway. |
| `PUBSUB_REDIS_URL` | Control-bus Redis the frontend publishes revocations to. |
| `USER_REGISTRATION_ENABLED` | Whether people can sign themselves up. See [Lock down self-registration](#lock-down-self-registration). |
| `ADMIN_USER_IDS` | Comma-separated better-auth user ids granted platform super-admin. See [Create the first admin](#create-the-first-admin). |

## Create the first admin

There are two separate kinds of "admin", set independently:

- **Platform super-admin:** put the better-auth user id(s) in `ADMIN_USER_IDS`
  on the frontend. A user signs up normally, then you add their id.
- **Admin WhatsApp number:** set `WHATSAPP_ADMIN_NUMBER` on the gateway. See
  below.

### The admin number

A deployment can name one number as its admin number through the gateway's
`WHATSAPP_ADMIN_NUMBER` setting. On boot, if that number is not already linked,
the gateway creates its session, prints a pairing code to its logs, and shows
the same code on the admin panel — so you can pair it without a browser.

The admin number is otherwise an ordinary session: it sends and receives like
any other. It also reserves a private command channel. Messages it receives that
start with the admin command prefix (`WHATSAPP_ADMIN_CMD_PREFIX`, default `am`)
are handled internally — they are **not** stored, emitted as events, or counted
against rate limits. In this release that channel is reserved but does nothing;
commands arrive in a later version.

## Hosting the services separately

The frontend can run anywhere it can reach MySQL and the router. The router and
gateway run near WhatsApp with the keystore volume. The router verifies callers
against the frontend's signing keys (JWKS) and the shared `apikey` table, so it
never calls back to the frontend on a request; the gateway verifies only the
router's assertion.

| Goal | What to do |
|---|---|
| Run the frontend on a separate host or serverless | Point `DATABASE_URL` at the shared MySQL (use a connection pooler on serverless) and `GATEWAY_URL`/`VITE_GATEWAY_URL` at the **router**. The browser talks to the router; the router reaches the gateway. |
| Bring your own MySQL/Redis | `deploy/docker-compose.external.yml` runs the app images against services you provide. |
| Router fetches signing keys from a different host | If the router fetches the frontend JWKS from a host other than `BETTER_AUTH_URL`, add that host to the frontend's `TRUSTED_ORIGINS`. |
| Add more gateways | Run another gateway with a unique `GATEWAY_ID`, its own persistent keystore volume, the same MySQL/Redis, and a `PUBLIC_URL` reachable by the router. The router will place new sessions on active gateways, but you must operate capacity and failover policy. |

## Health checks

Both Go services expose health endpoints:

| Service | Endpoint | Purpose |
|---|---|---|
| Router | `GET /healthz` | Liveness. |
| Router | `GET /readyz` | Readiness — checks MySQL and Redis. Use this for the router's container healthcheck. |
| Gateway | `GET /healthz` | Liveness. |
| Gateway | `GET /readyz` | Readiness — checks MySQL and Redis. Use this for the gateway's container healthcheck. |
| Gateway | `GET /metrics` | Prometheus metrics. |

## Running the gateway or router in Docker for development

Day to day you run the Go processes on the host (see
[Local development](/docs/dev/local-development)). To run one in a container
instead — same OS and toolchain as production, with the frontend still on the
host under Vite — use one of these compose profiles:

```bash
# Gateway, hot reload: air watches the mounted source and rebuilds in the container.
docker compose -f deploy/docker-compose.dev.yml --profile gateway-dev up -d --build

# Gateway, production image (rebuild to pick up code changes):
docker compose -f deploy/docker-compose.dev.yml --profile gateway up -d --build

# Router, production image (publishes :8090):
docker compose -f deploy/docker-compose.dev.yml --profile router up -d --build
```

Each profile reaches the host frontend's signing keys through
`host.docker.internal` (already allowed in `vite.config.ts`) and applies the WA
migrations on boot. The two gateway profiles both bind port 8080 — run only one
at a time.

Every profile reads the **same** `deploy/.env` the host processes use (copy it
from `deploy/.env.example`), so there is one place to set config. The compose
`environment:` block overrides only the variables that must differ inside a
container: `MYSQL_DSN` and `REDIS_URL` (the service hostnames),
`BETTER_AUTH_JWKS_URL` (the frontend is on the host), and `WHATSMEOW_STORE_DSN`
(a named volume). Change anything else in `deploy/.env`.

## Pairing a WhatsApp number

Pairing is the one step a deploy cannot script. Create a session (or set
`WHATSAPP_ADMIN_NUMBER`), open the QR or pairing code in the dashboard, and link
it from the phone's WhatsApp app. This uses an unofficial client — automated use
may violate WhatsApp's terms and risks a ban.
