# Dashboard (v2) — TanStack Start + better-auth

Realtime dashboard and identity core for the WhatsApp gateway. A **fullstack
TanStack Start** app (SSR + server functions, Vite plugin, Nitro server) with
**better-auth** for identity. Replaces the v1 React Router SPA.

The frontend server does only short-lived work: better-auth endpoints, minting
short-lived gateway JWTs, and direct MySQL **reads** via Drizzle for SSR/loaders.
It does **not** proxy gateway traffic — the **browser talks to the gateway
directly** (CORS + Bearer JWT) for both actions and the NDJSON event stream.

## Commands

```bash
pnpm install
pnpm dev            # vite dev server on :3000 (SSR + better-auth at /api/auth/*)
pnpm build          # vite build → .output (Nitro server bundle)
pnpm start          # run the built server (node .output/server/index.mjs)
pnpm typecheck      # tsc --noEmit (the router type tree is generated by build/dev)
pnpm test           # vitest (ndjson parser + cacheBridge event reducer)

pnpm auth:generate  # regenerate app/lib/db/auth-schema.ts from the better-auth config
pnpm db:migrate     # drizzle-kit migrate — apply the better-auth-OWNED auth tables
pnpm db:introspect  # drizzle-kit introspect — regen the READ-ONLY WA mirror (live DB)
pnpm gen:api        # regenerate app/lib/api/schema.d.ts from ../docs/openapi.yaml
```

`vite dev`/`vite build` run the TanStack Router code generator, which produces
`app/routeTree.gen.ts`. `tsc` consumes that generated tree, so run a build (or
dev) once before typechecking a fresh checkout.

## Auth & the gateway contract

better-auth is mounted at `/api/auth/*` (catch-all route `app/routes/api/auth.$.ts`).
Config in `app/lib/auth/server.ts`:

- **JWT** (jwt plugin): asymmetric **EdDSA/Ed25519**, `iss == aud == BETTER_AUTH_URL`,
  ~5 min expiry. JWKS at `/api/auth/jwks`, tokens minted at `/api/auth/token`.
  `definePayload` emits `sub` (user id), `activeOrganizationId`, `orgRole`
  (owner|admin|member for the active org), and `role` (platform admin role, e.g.
  `super_admin`). The gateway verifies these locally via JWKS.
- **API keys** (`@better-auth/api-key`): default hash = `base64url(SHA-256(rawKey))`
  unpadded. Keys are **org-scoped** via `references: "organization"`, so the
  owning org id is stored in the key's `referenceId` column. Create/list pass
  `organizationId` (the plugin maps it to `referenceId`); the gateway resolves
  the owning org by reading `apikey.referenceId`. Permissions map to
  `{read,send,manage,events}` under a single `wa` resource bucket.
- **Control bus** (publish-only): better-auth `after` hooks publish revocations to
  `PUBSUB_REDIS_URL` — `ctrl:apikey.revoked {keyId}` on key delete,
  `ctrl:user.banned {userId}` on ban, `ctrl:member.removed {userId,organizationId}`
  on member removal (`app/lib/server/control-bus.ts`). The frontend never touches
  the work Redis.

## Data ownership (§6.2, single-writer)

- **better-auth owns** the auth tables (user/session/account/verification/jwks/
  apikey/organization/member/invitation/twoFactor/admin) via the Drizzle adapter.
  Migrated from here with `drizzle-kit` (`app/lib/db/auth-schema.ts`).
- **The gateway owns** the WA app-data tables (wa_sessions, gateways, messages,
  chats, webhooks, outbox, event_log, whatsapp_*), migrated by the Go
  golang-migrate. The frontend models them as **READ-ONLY** Drizzle definitions
  (`app/lib/db/wa.ts`) and never writes them; SSR loaders read them directly.

## Architecture

- `app/lib/**` and `app/components/{ui,shell}/**` are the **shared boundary** —
  the gateway client, NDJSON stream consumer, `cacheBridge`, `eventBus`,
  query-key factory (`qk`), Zod schemas, generated OpenAPI types, and the
  better-auth client/server. Reused from v1, reshaped to Start idioms.
- Routes live under `app/routes/` as a file-based tree:
  - `_auth.tsx` + `_auth/*` — public auth surface (login/register/2fa).
  - `_app.tsx` — authenticated layout; resolves the session server-side in
    `beforeLoad` and puts it on the route context (`{ session }`).
  - `_app/admin/*` — super_admin surface (sessions/tenants/monitor/pairing).
  - `_app/user*` — user workspace (sessions/keys/webhooks) + the org switcher.
  - `_app/user.sessions.$sessionId.chats*` / `.contacts*` — viewer + contacts.
  - `index.tsx` — role-routed entry (→ /login, /admin/sessions, or /user/sessions).
  - Files prefixed `-` are colocated helpers excluded from the route tree.
